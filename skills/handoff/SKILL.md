# handoff — Session-to-session bootstrap doc generator

You create a `HANDOFF.md` at the repo root that lets a **fresh Claude Code
session** pick up exactly where the current one left off, without relying on
auto-compaction.

The fundamental insight: auto-compaction is lossy and unaware of repo state
outside the conversation. A deliberate handoff doc is **curated** (you decide
what matters), **grounded in current repo state** (git, architecture/,
Linear, .env.example), and **bootable** (it tells the next session exactly
which files to read and in what order — including the real code, not just
docs).

## HARD RULES — Read First

1. **Never silently overwrite an existing `HANDOFF.md`.** Always inspect it
   first. If it has the skill's own frontmatter (`generated_by: handoff-skill`),
   it's safe to overwrite — show the user the previous `generated_at` +
   `session_summary` and confirm. If it's a hand-crafted doc (no skill
   frontmatter), require explicit confirmation and offer the `--keep` flag
   alternative so the user doesn't lose work.
2. **Pull lessons from this session aggressively.** When the user corrected
   you, said "don't do X," or you saw something fail and pivoted, that's gold
   for the next session — put it in `## Things to AVOID`. Don't be timid; the
   next session benefits more from "user prefers X not Y" than from a
   sanitized happy-path summary.
3. **Tell the next session to read the actual code, not just docs.** Docs
   describe; code is. The next session must read the specific files we
   touched (with paths) before making changes, or it'll write code that
   doesn't match conventions, file structure, or in-flight refactors.
4. **The handoff is a snapshot, not a plan.** Capture what *is* (state of
   work, decisions made, things tried) — not what *should be done long-term*.
   That's what `architecture/plans.md` is for. The handoff's forward-looking
   section is one concrete next action, not a roadmap.
5. **Aim for ~150 lines.** A 400-line handoff is friction. Be terse and
   specific. Bullet points over paragraphs. File paths over prose.

## Decision Tree

```
What is the user asking?
├── "/handoff" or "create a handoff" / "hand off this session"
│   → DEFAULT flow: write HANDOFF.md at repo root (confirm before overwrite)
├── "/handoff --keep"
│   → KEEP flow: write HANDOFF-<YYYY-MM-DD-HHMM>.md, never overwrites
└── "/handoff <path>"
   → CUSTOM flow: write to that path
```

## DEFAULT Flow

### Step 1 — Verify location and check for existing HANDOFF.md

1. Confirm you're in a git repo: `git rev-parse --show-toplevel`. If not, ask
   the user where to write.
2. Check whether `HANDOFF.md` exists at the repo root. If it does:
   - Read its first 30 lines.
   - If frontmatter contains `generated_by: handoff-skill`: print
     > "Previous handoff: generated <generated_at>, summary: '<session_summary>'.
     > Overwrite? [y/N]"
   - If no skill frontmatter (looks hand-crafted): print
     > "`HANDOFF.md` exists and was not generated by this skill. Overwrite
     > would replace your hand-crafted doc. Options: (o)verwrite, (k)eep both
     > (write as HANDOFF-<timestamp>.md instead), (c)ancel?"
   - Wait for the user's choice before proceeding. Do NOT default to
     overwrite — silent overwrite of hand-crafted content is the most damaging
     thing this skill can do.

### Step 2 — Gather durable context (repo state outside the conversation)

Run these in parallel where possible:

```bash
git branch --show-current
git log -1 --format="%h %s (%ad)" --date=relative
git log --since="3 days ago" --pretty=format:"%ad %h %s" --date=short
git status --short -uall | head -50
git rev-parse --show-toplevel  # for project name
```

Read these files if they exist (skip silently if missing):

- `architecture/state.md` — TL;DR + stack + how-to-run + pointers
- `architecture/plans.md` — current roadmap (read so the handoff doesn't
  contradict it)
- `architecture/decisions.md` — material decisions the next session
  shouldn't re-litigate
- `README.md` — fallback if no architecture/
- `.env.example` — what credentials/env vars the next session needs
- `pyproject.toml` / `package.json` / `Cargo.toml` — language + key deps + run
  scripts
- `CLAUDE.md` (root and `.claude/`) — project-specific Claude instructions
  the next session must follow

Linear lookup (if the user has Linear MCP and a project resolves):

```python
# Try to match a Linear project from the git directory name or recent
# commit-message SOFT-NNNN references. If matched, list active issues
# (In Progress + Todo) and surface ones tied to this repo.
mcp__claude_ai_Linear__list_projects(query=<repo-name>)
mcp__claude_ai_Linear__list_issues(project=<id>, state="In Progress")
```

If no Linear project resolves, skip silently. Don't ask the user.

### Step 3 — Synthesize volatile context (from the current conversation)

This is the irreplaceable part. From the conversation history, distill:

- **What we worked on this session** — 3-8 concrete bullets, not a transcript.
- **Decisions made** — things you and the user agreed on that aren't in code
  yet (or are in code but worth flagging for the next session's mental model).
- **What was tried and rejected** — approaches you started down and pivoted
  away from. Include the *reason* — the next session will rediscover the
  rejected approach if it doesn't know why it failed.
- **Things to AVOID** — pull aggressively from:
  - User corrections ("no, don't do that", "stop", "use X instead")
  - User confirmations of non-obvious choices ("yes that was right" — means
    *keep doing it*, capture as a positive instruction)
  - Things that visibly failed and you had to recover from
  - Tool-use mistakes that wasted turns
- **Open questions waiting on user input.**
- **Suggested next concrete action** — the precise next change to make, with
  file paths if possible. ONE thing, not a roadmap.

### Step 4 — Compose the handoff

Use the template at `references/handoff-template.md` as the structure. Fill
each section from durable + volatile context.

The `## START HERE — Reading order` section is the most important. It must:

1. Tell the next session to **read this entire HANDOFF.md first**.
2. List **specific files** to read next, in order, with **one-line "why"
   per file**. Group as:
   - Architecture / docs (state.md, plans.md, decisions.md, README)
   - **Actual code files we touched this session** — with paths (see
     HARD RULE 3).
   - Test files for that code if they exist.
3. List **verification commands** to run before making any change (test
   suite, type check, lint — whatever the repo uses).

### Step 5 — Self-add the skill frontmatter

Prepend this YAML frontmatter to the handoff so future runs can recognize
it as safe-to-overwrite:

```yaml
---
generated_by: handoff-skill
generated_at: <ISO-8601 UTC>
session_summary: <one-line, no quotes inside>
project: <repo dir name>
branch: <git branch>
last_commit: <short-sha>
---
```

### Step 6 — Write the file and print the next-session opener

Write to `HANDOFF.md` at repo root (or wherever Step 1 confirmed). Then
print to the user:

> Wrote `HANDOFF.md` (~N lines). To bootstrap the next session, open a fresh
> Claude Code conversation in this directory and paste:
>
> > Read `HANDOFF.md` and follow the START HERE reading order before doing
> > anything else. Then summarize what we left off on and propose your first
> > move.

This opening line is load-bearing — it tells the next session to read
*before* acting, and to **summarize then propose** rather than just diving
in (the user gets a checkpoint to redirect if the next session misread
the handoff).

## KEEP Flow

Same as DEFAULT, but Step 1 writes to `HANDOFF-<YYYY-MM-DD-HHMM>.md` and
never touches the existing `HANDOFF.md`. Tell the user the new path in
Step 6 so they know which to point the next session at.

## CUSTOM Flow

Same as DEFAULT, but Step 1's path is the user-specified one. Still apply
the existing-file overwrite check at that path.

## What goes in the conversation summary vs the volatile bullets

This is the line that separates "useful handoff" from "noise." Use this
rubric:

- **In `## What we worked on`:** the user-visible outputs of the session.
  Commits landed, files created/modified, decisions agreed. Things you'd
  put in a standup.
- **NOT in the handoff:** exploration paths, tool-call sequences, thinking
  out loud, things you considered then rejected without acting on. The
  next session doesn't need a replay; it needs the deltas.
- **In `## Things to AVOID`:** explicit user feedback + observed failures
  the next session would otherwise re-encounter. *"User prefers terse
  responses, no trailing summaries"* > *"I was verbose at first."*
- **In `## Decisions made (not yet in code)`:** agreements reached in
  conversation that haven't been implemented. Critical because the next
  session would otherwise re-debate them with the user.

## Edge cases

- **No architecture/ folder + no README:** the handoff can still be useful —
  pull from the conversation itself + git log. Note in the START HERE
  section that the project lacks structured docs.
- **Brand-new repo, no commits:** report that under `## Current state of
  work` (`Last commit: (none yet)`); the handoff is still useful for
  capturing what we discussed and planned.
- **Multi-day session that's already been compacted:** flag this in the
  `## What we worked on` section. The model can only see the
  post-compaction transcript; the user may need to add prior context
  manually. This is one of the limits of any handoff — it can only capture
  what's in the current conversation.
- **No Linear MCP / no project resolves:** silently skip the "Active
  workstreams" section. Don't insert "(no Linear project found)" filler —
  just omit.
- **Sensitive content in conversation:** never write credentials, API keys,
  tokens, or other secrets into the handoff. If the user discussed
  credentials, write a placeholder like "*[credential discussed in
  session — do not include in handoff]*" and warn them.
